# qwen3-pyannote-diar-asr 仕様書

## 1. 概要

本アプリは、Docker / Windows 11 環境で動作する日本語音声文字起こしツールである。  
主な目的は、単一の長時間音声ファイルを入力として、

1. 音声を 16kHz mono WAV に変換し
2. pyannote により話者分離を行い
3. Qwen3-ASR により各セグメントを文字起こしし
4. 話者ラベル付きの VTT / TXT を出力する

ことである。

本アプリは、研究インタビューや聞き取り音声の一次文字起こしを主用途とする。  
最終成果物の品質は本アプリ単体で完結させることを前提とせず、必要に応じて Zoom VTT との突合や LLM による文脈補正を後段で行う構成を想定する。

---

## 2. 設計方針

### 2.1 基本方針
本アプリは、ASR 単体の極限的な精度追求よりも、以下を重視する。

- 長時間音声でも完走しやすいこと
- 例外時に原因追跡しやすいこと
- GPU メモリ使用を明示的に制御できること
- Qwen3-ASR への置き換え後も構成が単純であること
- 後段の Zoom VTT 突合・LLM 修正と競合しないこと

### 2.2 アプリの位置づけ
本アプリは、話者分離付きの一次文字起こし生成ツールである。  
固有名詞の最終補正、文脈理解を要する補完、表記統一などは、原則として後段処理に委ねる。

### 2.3 シンプル化方針
Qwen3-ASR 版では、従来の Kotoba-Whisper 版に存在した辞書 Layer A 関連の複雑な制御を持ち込まない。  
ASR 前段での prompt 制御や token budget 管理は行わず、基本動作を明確に保つことを優先する。

## 3. 想定動作環境

### 3.1 OS / 実行環境
- Windows 11
- Docker Desktop
- NVIDIA GPU 利用環境
- `docker run --gpus all` で起動

### 3.2 主なソフトウェア構成
- Python 3.11 系
- PyTorch
- `pyannote.audio`
- `transformers`
- `huggingface_hub`
- `pydub`
- `ffmpeg`
- `tqdm`

### 3.3 主なモデル
- 話者分離: `pyannote/speaker-diarization-3.1`
- ASR: Qwen3-ASR 系モデル
- 初期実装対象: `Qwen/Qwen3-ASR-1.7B`

### 3.4 モデルの役割分担
- pyannote: 誰が話したかを区切る
- Qwen3-ASR: 各区間で何が話されたかを文字起こしする

Qwen3-ASR 自体は話者分離モデルではないため、話者ラベル付き出力には pyannote を併用する。

---

## 4. 入出力仕様

### 4.1 入力
入力は音声ファイル 1 本である。  
想定する主な入力形式は m4a だが、ffmpeg で読み込める音声形式であれば拡張可能である。

指定方法:
- `INPUT_FILENAME` 環境変数でファイル名を指定
- 実体は `WORK_SOURCE` 配下に置く

### 4.2 出力
出力は `WORK_OUTPUT` 配下に保存する。  
`WORK_OUTPUT` が未指定の場合は `WORK_SOURCE/output` を用いる。

出力ファイル:
- `<base>.vtt`
- `<base>.txt`

必要に応じて将来追加しうる補助ログは、通常出力とは分離して扱う。

### 4.3 VTT 出力
VTT は話者ラベル付きで出力する。  
各セグメントは `<v speaker>` 形式で出力する。

### 4.4 TXT 出力
TXT は 1 行 1 セグメントの簡易出力とし、各行の先頭に話者ラベルを付す。

## 5. 処理フロー

### 5.1 概要
処理フローは以下の通りである。

1. 入力ファイル存在確認
2. ffmpeg による 16kHz mono WAV 変換
3. pyannote 話者分離モデルのロード
4. 話者分離の実行
5. 話者分離用リソースの解放
6. Qwen3-ASR モデルまたは pipeline のロード
7. セグメント単位 ASR
8. VTT / TXT 書き出し
9. 一時ディレクトリ削除

### 5.2 音声変換
入力音声は ffmpeg を subprocess で呼び出し、16kHz mono WAV に変換する。  
変換後の WAV は一時ディレクトリに保存する。

### 5.3 話者分離
pyannote により話者分離を行う。  
`NUM_SPEAKERS` が指定されていれば固定話者数として扱い、未指定なら自動推定とする。

### 5.4 ASR
話者分離で得られた各セグメントを WAV として切り出し、Qwen3-ASR に渡す。  
ASR はセグメントごとに逐次実行し、必要以上に複雑な前処理は行わない。

### 5.5 後段処理との分離
本アプリ内では、文脈依存の高度補正や固有名詞辞書による強制補正を必須仕様としない。  
必要な補正は後段の LLM や人手確認に委ねる。

---

## 6. GPU メモリ管理

### 6.1 方針
話者分離後は、ASR 実行前に不要になった話者分離リソースを解放し、GPU メモリを明示的に整理する。

### 6.2 実施内容
- 話者分離オブジェクトへの参照を外す
- `gc.collect()` を呼ぶ
- CUDA 利用時は `torch.cuda.empty_cache()` を呼ぶ
- 解放前後の VRAM 状況をログに出す

### 6.3 目的
- pyannote と Qwen3-ASR の連続使用による VRAM 圧迫を緩和する
- 4060 Ti 16GB クラスでも安定運用しやすくする
- メモリエラー時の切り分けを容易にする

---

## 7. Qwen3-ASR 実装方針

### 7.1 基本
Qwen3-ASR 版では、Kotoba-Whisper 版に存在した Whisper 固有の prompt 制御に依存しない。  
そのため、ASR 実装は Qwen3-ASR に対して素直な入力を与える構成を基本とする。

### 7.2 実装原則
- Qwen3-ASR の標準的な推論経路を優先する
- Whisper 固有の `prompt_ids` 前提コードを引き継がない
- token budget のような辞書投入制御を持ち込まない
- モデル依存の特殊処理は最小限にとどめる
- まずは安定完走を優先し、その後に必要最小限の改善を検討する

### 7.3 逐次処理
長時間音声への対応のため、セグメントごとに処理する。  
一括処理よりも、失敗箇所の切り分けとメモリ管理を優先する。

### 7.4 失敗時の考え方
失敗時は、Qwen3-ASR の読み込み、入力音声、セグメント切り出し、GPU メモリ、transformers まわりの整合性を優先的に確認する。  
Whisper 版由来の fallback 機構には依存しない。

## 9. ログ方針

### 9.1 コンソールログ
コンソールには以下を出力する。

- 入力確認
- ffmpeg 変換開始・終了
- 話者分離開始・終了
- GPU メモリ解放開始・終了
- ASR 開始・進捗・終了
- 出力ファイル保存先
- 例外時の traceback

### 9.2 ログの目的
- 長時間処理の進行状況を確認しやすくする
- 問題発生時の切り分けを容易にする
- モデル切り替え時の比較をしやすくする

---

## 10. 運用方針

### 10.1 通常運用
通常運用では、話者分離 + Qwen3-ASR + VTT/TXT 出力の最小構成を基準とする。

### 10.2 後段補正
固有名詞の補正、文脈補正、表記統一は後段で行う。  
本アプリは一次転記の安定生成に集中する。

### 10.3 比較運用
改修後の評価では、少なくとも以下を比較対象とする。

1. 旧版 Kotoba-Whisper 実装
2. 新版 Qwen3-ASR 実装
3. 必要に応じて Zoom VTT 併用後の結果

---

## 11. 非目標

本アプリは以下を現時点の目標としない。

- Qwen3-ASR 単体で話者分離を完結させること
- Whisper 固有の `prompt_ids` を Qwen3-ASR に流用すること
- token budget により辞書語を選別して投入すること
- Layer A fallback のような Whisper 版特有の制御を維持すること
- 文脈理解を伴う高度補正を ASR 前段で完結させること

これらは、本アプリの範囲外とする。

## 13. まとめ

本アプリは、pyannote による話者分離と Qwen3-ASR による文字起こしを組み合わせ、話者ラベル付きの一次文字起こしを安定して生成することを目的とする。  
Qwen3-ASR 版では、Whisper 版由来の複雑な Layer A 系制御を持ち込まず、構成を単純化する。  
これにより、保守性・可読性・安定性を高め、後段の LLM 補正や人手確認と役割分担しやすい構成を実現する。
